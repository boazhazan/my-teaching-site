יחידה 3: ASP.NET Core Razor Pages

ניהול מצב (Session) ואבטחה

Statelessness, אובייקט Session, מערכת Login/Logout ואבטחת דפים

1. בעיית ה-Statelessness

פרוטוקול HTTP הוא פרוטוקול Stateless — כלומר, חסר מצב. כל בקשה (Request) שנשלחת לשרת היא עצמאית לחלוטין. השרת לא "זוכר" שום דבר מהבקשה הקודמת.

מבחינת השרת, כל בקשה מגיעה מלקוח זר. גם אם לפני שנייה שלחת בקשה והשרת החזיר לך תשובה — בבקשה הבאה, הוא לא יודע מי אתה.

💡 אנלוגיה — מוכר בחנות

דמיינו מוכר בחנות שסובל מאובדן זיכרון לטווח קצר. כל פעם שאתם ניגשים אליו, הוא לא מזהה אתכם — גם אם יצאתם מהחנות לפני רגע. כדי להוכיח מי אתם, תצטרכו להציג תעודת זהות בכל פעם מחדש. ככה בדיוק HTTP עובד.

זה יוצר בעיות משמעותיות:

  • איך לשמור שמשתמש מחובר? — אחרי שהמשתמש מזין שם וסיסמה, איך נזכור שהוא כבר אומת?
  • איך לזכור מה בעגלת הקניות? — אם כל בקשה עצמאית, העגלה "נמחקת" בין דף לדף
  • איך להתאים תוכן אישי? — אם לא יודעים מי המשתמש, אי אפשר להציג לו תוכן רלוונטי

שתי בקשות מאותו דפדפן — השרת רואה אותן כלקוחות שונים

🌐 דפדפן
בקשה #1: "הצג פרופיל"
→
🖥️ שרת
מי אתה? 🤷
🌐 דפדפן
בקשה #2: "הוסף לעגלה"
→
🖥️ שרת
מי אתה?? 🤷
השרת לא יודע ששתי הבקשות הגיעו מאותו משתמש!

הפתרון? אנחנו צריכים מנגנון שישמור מידע על המשתמש בין בקשות. וזה בדיוק מה ש-Session עושה.

2. אובייקט Session

Session הוא מנגנון שמאפשר לשרת לשמור נתונים בזיכרון השרת, עבור כל משתמש בנפרד.

איך זה עובד?

  1. כשמשתמש מגיע לאתר בפעם הראשונה, השרת יוצר Session ID ייחודי
  2. ה-Session ID נשמר אצל המשתמש כ-Cookie (קובץ טקסט קטן בדפדפן)
  3. בכל בקשה עתידית, הדפדפן שולח את ה-Cookie אוטומטית
  4. השרת קורא את ה-Session ID מה-Cookie, ויודע איזה נתונים שייכים למשתמש הזה

איך Session עובד

🌐 דפדפן
בקשה + Cookie: ID=abc123
→
🖥️ שרת
abc123 → {user: "בועז"}
השרת מזהה את המשתמש בזכות ה-Session ID שבתוך ה-Cookie

הגדרת Session ב-ASP.NET Core

כדי להשתמש ב-Session, צריך להוסיף כמה שורות לקובץ Program.cs:

// Program.cs - הגדרה var builder = WebApplication.CreateBuilder(args); builder.Services.AddRazorPages(); builder.Services.AddDistributedMemoryCache(); // נדרש ל-Session builder.Services.AddSession(); // הוספת שירות Session var app = builder.Build(); app.UseStaticFiles(); app.UseRouting(); app.UseSession(); // הפעלת Session middleware app.MapRazorPages(); app.Run();

שמירה וקריאה מ-Session

בתוך דפי Razor (קבצי .cshtml.cs), משתמשים בפקודות הבאות:

// שמירה ל-Session HttpContext.Session.SetString("username", "בועז"); HttpContext.Session.SetInt32("age", 25); // קריאה מ-Session string name = HttpContext.Session.GetString("username"); // "בועז" int? age = HttpContext.Session.GetInt32("age"); // 25 // מחיקת ערך ספציפי HttpContext.Session.Remove("username"); // מחיקת כל ה-Session HttpContext.Session.Clear();

דוגמה מלאה: שמירת שם משתמש

// בדף .cshtml.cs public class WelcomeModel : PageModel { public string UserName { get; set; } public void OnPost() { string username = Request.Form["username"]; HttpContext.Session.SetString("user", username); } public void OnGet() { string user = HttpContext.Session.GetString("user"); if (user == null) { UserName = "אורח"; } else { UserName = user; } } }
💡 חשוב לזכור

Session נמחק כשהמשתמש סוגר את הדפדפן או אחרי timeout (ברירת מחדל: 20 דקות ללא פעילות). אם צריך לשמור מידע לטווח ארוך, צריך בסיס נתונים.

3. מערכת Login/Logout

עכשיו שאנחנו מבינים איך Session עובד, בואו נבנה מערכת כניסה (Login) ויציאה (Logout) שלמה.

תהליך Login — שלב אחרי שלב

1
המשתמש ממלא שם וסיסמה בטופס Login
2
הטופס נשלח לשרת בבקשת POST
3
השרת בודק האם שם המשתמש והסיסמה נכונים
4
אם נכון — השרת שומר את השם ב-Session
5
השרת מפנה (Redirect) את המשתמש לדף הראשי
6
הדפדפן מציג את הדף הראשי עם "שלום [שם]"

דף Login — קוד מלא

// Login.cshtml.cs public class LoginModel : PageModel { public string ErrorMessage { get; set; } public void OnGet() { // הצגת טופס ההתחברות } public void OnPost() { string user = Request.Form["username"]; string pass = Request.Form["password"]; if (user == "admin" && pass == "1234") { HttpContext.Session.SetString("user", user); Response.Redirect("/Index"); } else { ErrorMessage = "שם משתמש או סיסמה שגויים"; } } }

דף Login — טופס ה-HTML

<!-- Login.cshtml --> @page @model LoginModel <h2>התחברות</h2> <form method="post"> <label>שם משתמש:</label> <input type="text" name="username" required /> <label>סיסמה:</label> <input type="password" name="password" required /> <button type="submit">כניסה</button> </form> @if (!string.IsNullOrEmpty(Model.ErrorMessage)) { <p style="color:red">@Model.ErrorMessage</p> }

Logout — ניקוי ה-Session

כשהמשתמש לוחץ "התנתק", מנקים את ה-Session ומפנים לדף הבית:

// Logout.cshtml.cs public class LogoutModel : PageModel { public void OnGet() { HttpContext.Session.Clear(); // מחיקת כל נתוני ה-Session Response.Redirect("/Index"); // חזרה לדף הראשי } }

תפריט דינמי — "התחבר" או "שלום [שם]"

ניתן להציג תפריט שונה למשתמש מחובר ולמשתמש אורח:

<!-- _Layout.cshtml (תבנית האתר) --> @if (Context.Session.GetString("user") != null) { <span>שלום @Context.Session.GetString("user")!</span> <a href="/Logout">התנתק</a> } else { <a href="/Login">התחבר</a> }
✅ סיכום — Login/Logout

Login: בדיקת שם+סיסמה → שמירה ב-Session → Redirect לדף הראשי.
Logout: ניקוי Session → Redirect לדף הבית.
תפריט: בדיקה אם יש ערך ב-Session → הצגת "שלום" או "התחבר".

4. אבטחת דפים

עכשיו שיש לנו מערכת Login, אנחנו צריכים להגן על דפים שמיועדים רק למשתמשים מחוברים. למשל: דף פרופיל, דף הזמנות, לוח בקרה.

בדיקה בתחילת OnGet

הדרך הפשוטה ביותר: בכל דף מוגן, לבדוק בתחילת OnGet האם יש Session פעיל. אם אין — להפנות לדף Login:

// Profile.cshtml.cs — דף מוגן public class ProfileModel : PageModel { public string UserName { get; set; } public void OnGet() { if (HttpContext.Session.GetString("user") == null) { Response.Redirect("/Login"); return; } // המשתמש מחובר — להמשיך כרגיל UserName = HttpContext.Session.GetString("user"); } }

הפרדת הרשאות: Admin vs User

לפעמים רוצים להבדיל בין משתמש רגיל למנהל (Admin). ניתן לשמור את התפקיד ב-Session:

// בזמן Login — שמירת התפקיד if (user == "admin") { HttpContext.Session.SetString("role", "admin"); } else { HttpContext.Session.SetString("role", "user"); } // בדף ניהול — בדיקת הרשאה public void OnGet() { string role = HttpContext.Session.GetString("role"); if (role != "admin") { Response.Redirect("/AccessDenied"); return; } // רק מנהל מגיע לכאן }
⚠️ אזהרת אבטחה חשובה!

אל תשמרו סיסמאות בטקסט גלוי! בדוגמאות כאן השתמשנו ב-"1234" לצורך הלמידה, אבל בפרויקט אמיתי חובה להצפין (hash) סיסמאות לפני שמירה בבסיס הנתונים. ספריות כמו BCrypt עושות את זה בקלות.

💡 סיכום — 4 עקרונות אבטחה בסיסיים

1. בדיקת Session בכל דף מוגן — אם אין, redirect ל-Login
2. הפרדת הרשאות — Admin ו-User לא רואים אותם דפים
3. הצפנת סיסמאות — לעולם לא לשמור סיסמה כטקסט רגיל
4. ניקוי Session — ב-Logout חובה לעשות Clear()

בדיקת הבנה

בחנו את עצמכם — ענו על השאלות הבאות כדי לוודא שהבנתם את החומר:

שאלה 1: למה HTTP נחשב Stateless?

  • א) כי הוא לא מצליח לשמור קבצים
  • ב) כי כל בקשה היא עצמאית — השרת לא זוכר בקשות קודמות
  • ג) כי הוא עובד רק עם HTML סטטי
  • ד) כי הוא דורש חיבור קבוע לאינטרנט

שאלה 2: מה יוצג?

נתון הקוד הבא. אם המשתמש כבר התחבר בשם "דני", מה יוצג?

public void OnGet() { string name = HttpContext.Session.GetString("user"); if (name != null) WelcomeMessage = "שלום " + name + "!"; else WelcomeMessage = "אורח/ת יקר/ה, אנא התחבר/י"; }
  • א) "אורח/ת יקר/ה, אנא התחבר/י"
  • ב) "שלום דני!"
  • ג) "שלום null!"
  • ד) שגיאה — name לא מוגדר

שאלה 3: השלימו את הקוד

השלימו את ה-OnPost כך שישמור שם משתמש ב-Session אחרי Login מוצלח:

public void OnPost() { string user = Request.Form["username"]; string pass = Request.Form["password"]; if (CheckPassword(user, pass)) { ("user", user); Response.Redirect("/Index"); } }

שאלה 4: סדרו את תהליך ה-Login

מה הסדר הנכון של השלבים בתהליך Login?

  1. המשתמש ממלא שם וסיסמה בטופס
  2. הטופס נשלח לשרת (POST)
  3. השרת בודק שם וסיסמה
  4. השרת שומר את השם ב-Session
  5. השרת מפנה (redirect) לדף הראשי
  6. הדפדפן מציג את הדף הראשי עם "שלום [שם]"

בחרו את הסדר הנכון:

  • א) 1 → 2 → 3 → 4 → 5 → 6
  • ב) 1 → 3 → 2 → 4 → 6 → 5
  • ג) 2 → 1 → 3 → 5 → 4 → 6
  • ד) 1 → 2 → 4 → 3 → 5 → 6
